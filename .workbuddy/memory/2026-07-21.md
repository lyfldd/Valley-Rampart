## 存档系统设计文档

- 产出：`河谷防线开发计划书具体内容/存档系统设计文档.md`
- 背景：阿铁要做存档系统（SO 做配置 + JSON 做存档），询问设计是否有问题
- 核心结论：
  - `UnitDataManager` 名不副实——它加载只读 SO 配置，是 Config Catalog 不是 Data Manager，建议改名 `UnitConfigCatalog`（`UnitData` → `UnitConfig`，`RulerData` → `RulerConfig`）
  - 加载配置（Resources/asset）≠ 加载存档（persistentDataPath/JSON），两者并行分开
  - 三层架构：POCO 存档数据层（GameSaveData/UnitSaveData/RulerSaveData/TimeSaveData）→ SaveManager（保存/加载/删除/版本迁移，原子写入）→ JsonSaveSerializer（封装 JsonUtility）
  - 读档流程：清空场景 → 恢复时间/资源 → 遍历存档单位「先 SpawnUnit 走默认初始化，再 RestoreFromSave 覆盖」
  - 需给 UnitController/RulerController/TimeManager 加 Restore* 方法（TimeManager.RestoreState 直接赋值不发事件）
- 坑点已写入文档：JsonUtility 不支持 Dictionary/Vector3/多态、saveVersion 从第一天就要加

## 存档系统 v2 · ISaveable 架构

- 产出：`河谷防线开发计划书具体内容/存档系统设计文档_v2_ISaveable架构.md`
- 触发：阿铁指出 v1 的 SaveManager.CaptureSnapshot 是硬编码，每加系统都要改 SaveManager，违反开闭原则。他自己研究了 ISaveable 接口契约模式（工业级标准），问评估
- 结论：阿铁的方向完全正确，但他贴的方案有 4 个坑：
  1. `Dictionary<string, object>` 不能用 JsonUtility 序列化 → 改用 `List<ModuleSaveEntry>`，每模块数据序列化成字符串
  2. `object` 多态反序列化拿不回类型 → 引入 `SavePayload`（typeName + json + version）自带类型信息
  3. 动态对象（UnitController）注册时机 → 引入 `ISaveableSpawner`，UnitFactory 实现，加载时先 Spawn 再 LoadState
  4. 模块间加载依赖（TimeManager 必须先于单位恢复）→ 引入 `SaveLoadPhase.Global / Scene` 两阶段
- 关键决策：
  - SaveId 用 GUID 保证唯一（同类单位多个，不能共用 ID）
  - RulerController.LoadState 不再 SpawnMonarch，君主走 UnitController ISaveable 流程，避免状态不一致
  - UnitController.Die 必须先 UnregisterSaveable 再 Destroy，防 MissingReferenceException
  - 模块版本号 SavePayload.version 独立于全局 saveVersion，每个模块自治迁移
  - 天然兼容：新增模块旧存档没有条目则跳过；删除模块旧存档有条目则 warning 跳过
- v1 文档已加 deprecated 标注，但 POCO 数据结构/原子写入/命名重构建议仍有效
