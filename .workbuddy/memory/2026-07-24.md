## 销毁架构薄弱点诊断（阿铁主动反思）

- 触发：阿铁问"加载很到位了，但这次 bug 是不是说明销毁相关还很薄弱"
- 结论：**是的，系统性薄弱**。销毁散在至少 6 处（UnitController.Die / RulerController.DestroyAllSceneUnits/DestroyMonarchForMenuReturn/RemoveDuplicateMonarchs / PausePanel.OnQuitClicked / GameOverPanel.OnBackToMenuClicked / GameBootstrap.Awake 的事后清理），无统一协调者，对比 LoadManager 的缺失明显
- 5 个薄弱点：
  1. 没有销毁协调者（加载有 LoadManager 4 阶段调度，销毁各组件自己管自己）
  2. 返回主菜单两条路径不一致：PausePanel 完整（Save+DestroyMonarch+DisableInput+LoadScene），GameOverPanel 只 LoadScene（Bug 6 根源）
  3. 被动清理代替主动销毁：CleanupDestroyedSaveables / UnitRegistry.Clear 都是事后扫描兜底
  4. DontDestroyOnLoad 清单不透明，靠人记（拆 RulerController 就踩了这个坑）
  5. Singleton 退出保护不完整（_isQuitting 关 Domain Reload 不重置，Bug 1）
- 务实建议（给阿铁）：**别急着建统一销毁系统**。当前就君主+几个 Manager，复杂度未到需 TeardownManager 的程度；过早建框架会锁死结构。加载需统一是因时序+依赖+缓存复杂，销毁还没到那个量级
- 现在值得补的 2 件（低成本高收益）：
  1. 抽统一"返回主菜单"出口，PausePanel 和 GameOverPanel 都调它，消除不一致
  2. Singleton 加 [RuntimeInitializeOnLoadMethod(SubsystemRegistration)] 重置 static（Domain Reload 防护）
- 等内容系统（敌人/建筑/物品）铺开再考虑：完整销毁协调者 / DontDestroyOnLoad 注册表 / 主动通知注册表

## 销毁系统设计文档生成（阿铁改主意：现在就做）

- 触发：阿铁说"等到以后多了系统就不好了，反正是多任务并行，现在就做统一销毁系统"，要求写文档，特别强调销毁顺序
- 先确认关键事实：UnitFactory.SpawnUnit（行78）用 Instantiate 没设 DontDestroyOnLoad → 拆分 RulerController 后君主单位已不是 DontDestroyOnLoad，LoadScene 会自动销毁，但 UnitController 无 OnDestroy 反注册 → 注册表残留需事后清理（隐患）
- 产出：`河谷防线开发计划书具体内容/2.4_销毁系统设计.md`（新建，完整可落地）
- 设计核心：
  - TeardownManager（Singleton）对称于 LoadManager，管销毁时序
  - 三级 API：TeardownScene（读档/新游戏前）/ TeardownForReturnToMenu（返回主菜单）/ TeardownForQuit（退出）
  - 销毁顺序按反拓扑序：InputManager.DisableInput → UI Panels → UnitController 们 → RulerController 反订阅 → TimeManager/DifficultyManager/WorldSystem → SaveManager → UnitRegistry.Clear → SaveManager 清理 → EventBus.Clear（最后）
  - 三个陷阱：Destroy 延迟 / OnDestroy 时序不可靠不能做关键清理 / DontDestroyOnLoad 盲区
  - 不引入 ITeardownable 接口，硬编码顺序（Manager 数量少，简单直接）
  - Core Manager（GameStateManager/InputManager/SaveManager/LoadManager）跨场景保留，业务 Manager Teardown 重置状态
- Level 2（返回主菜单）关键决策：不调 EventBus.Clear()，只让业务 Manager 显式 Unsubscribe，Core Manager 订阅保留（避免重新 Subscribe 的麻烦）。EventBus.Clear 只在 Level 3（退出）调
- 实施清单：新增 2 个（TeardownManager.cs + TeardownPhase）+ 修改 7 个（RulerController/TimeManager/DifficultyManager/WorldSystem/SaveManager 加 Teardown() + PausePanel/GameOverPanel 改调 TeardownManager）+ 附带 Singleton 加 Domain Reload 重置
- 待阿铁确认 3 点：业务 Manager Teardown 要不要重置状态 / SaveManager 的 GameStateChangedEvent 订阅保留与否 / 要不要引入 ITeardownable 接口

## 销毁系统设计修正：阿铁质疑销毁必要性 → 发现过度设计

- 触发：阿铁问"返回主菜单但没退出游戏，哪些该保留哪些该销毁"
- 关键发现：Grep 场景文件确认**所有 Manager（9个）都挂在 MainMenuScene 的独立 GameObject 上**（DontDestroyOnLoad）。项目目前只有 MainMenuScene 一个场景，没有 GameScene。WorldSystem 和 DifficultyManager 不在场景里（可能还没加，会触发空壳创建）
- 核心修正：**业务 Manager 只需 ResetState（重置状态），不需要反订阅、不需要反注册 ISaveable**——因为 Manager 都保留（DontDestroyOnLoad，返回主菜单后全都活着），订阅和注册都要继续用。原方案的"反订阅+反注册+重置"是过度设计
- 保留 vs 销毁决策表（已写入文档 5.2 节）：
  - Core Manager：保留，状态不动
  - 业务 Manager：保留，ResetState 重置运行时状态
  - 场景内 UnitController + 君主单位：主动反注册+销毁
  - UI Panels：LoadScene 自动销毁
  - EventBus：不 Clear（Manager 都保留，订阅也保留）
  - SaveManager._saveables：不清空（业务 Manager 的 ISaveable 注册保留）
- 文档已更新（4 处 Edit）：
  - 5.2 节重写：加决策表 + 修正流程（ResetState 代替 Teardown）+ 解释为什么不反订阅/不反注册
  - 4.3 枚举：ManagerUnsubscribed → ManagerReset，删 UIUnsubscribed
  - 5.3 Level 3 简化：不调 Manager.Teardown，依赖 Unity OnDestroy + EventBus.Clear（Unsubscribe 对空字典是 no-op，顺序不敏感）
  - 6.1 接口：ITeardownable → IResettable（只有 ResetState）
  - 实施清单：业务 Manager 加 ResetState() 而非 Teardown()，SaveManager 不新增方法
- 待确认点更新：业务 Manager ResetState 要不要彻底重置所有字段（vs 只停止运行让 InitializeWorld 覆盖）
- 附带发现：WorldSystem/DifficultyManager 不在 MainMenuScene，需确认是否漏挂
