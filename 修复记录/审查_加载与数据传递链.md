# 加载链路与数据传递链 审查报告

> 日期：2026-07-21
> 范围：新建游戏 + 读档游戏 两条链路的稳定性
> 触发：君主系统修复后，对"双份数据"与"数据传递丢失"两类风险做系统性排查
> 涉及文件：`Singleton.cs`、`GameBootstrap.cs`、`RulerController.cs`、`UnitController.cs`、`UnitFactory.cs`、`SaveManager.cs`、`UnitRegistry.cs`

---

```
              └─ BindExistingMonarch
```

## 二、风险点详细分析

### R1: Singleton 自动创建副作用（隐式分离）

**位置**：`Singleton<T>.Instance` getter

**现象**：
```csharp
if (_instance == null)
{
    _instance = FindObjectOfType<T>();
    if (_instance == null)
    {
        GameObject go = new GameObject($"[Singleton] {typeof(T).Name}");
        _instance = go.AddComponent<T>();  // ← 触发 Awake
        DontDestroyOnLoad(go);
    }
}
```
任何代码访问任意未初始化的 Singleton 都会触发 `AddComponent<T>` 创建空 GameObject。

**影响**：
- 任意 Manager 都可能被"隐式自动创建"，且实例上 SerializeField 引用全为 null
- 自动创建的实例会执行 Awake → 订阅事件 + RegisterSaveable
- 这种实例与场景中真正配置的对象是分离的两个对象，难以排查

**触发条件**：任何代码在 Manager 真正初始化前访问 `XxxManager.Instance`

**修复建议**：
- 短期：在 Instance getter 的自动创建分支加 `Debug.LogWarning`，监控生产环境的隐式创建
- 长期：考虑禁止自动创建，改为抛异常或返回 null，强制开发者在场景中显式放置

---

### R2: 场景预置的非君主单位读档后会双份【高风险】

**位置**：`GameBootstrap.InitializeScene` 读档分支 → `RulerController.DestroyAllSceneMonarchs`

**现象**：`DestroyAllSceneMonarchs` 只清理带 `PlayerInputHandler` 的单位：
```csharp
foreach (var unit in allUnits)
{
    if (unit.GetComponent<PlayerInputHandler>() == null) continue;  // ← 非君主单位跳过
    ...
}
```

**影响**：
- 开发期场景里常预置 NPC/敌人（带 UnitController 但无 PlayerInputHandler）用于测试
- 读档时这些预置单位不会被清理
- `SaveManager.Load` 阶段 1.5 的 `SpawnFromSave` 又会创建存档版本
- **结果：场景中出现 2 份相同单位**

**触发条件**：场景中预置了非君主单位（开发期常见，正式版理论上不会但要防御）

**修复建议**：
- 把 `DestroyAllSceneMonarchs` 改名/扩展为 `DestroyAllSceneUnits`，清理所有场景预置的 `UnitController`
- 或新增独立的 `ClearSceneUnits()` 方法，在场景切换/读档前调用
- 这是用户最担心的"双份数据"风险点，优先级 P0

---

### R3: 存档重复 SaveId 会导致双份 GameObject【中风险】

**位置**：`UnitFactory.SpawnFromSave`

**现象**：`SpawnFromSave` 不检查 `_saveables` 是否已有该 SaveId：
```csharp
public void SpawnFromSave(ModuleSaveEntry entry)
{
    ...
    GameObject go = SpawnUnit(config, pos);  // 创建实例 + Initialize 注册新 GUID
    if (go != null)
    {
        controller.OverrideSaveId(entry.saveId);  // 换为存档 ID
    }
}
```

**影响**（假设存档里有两个相同 SaveId 的 entry）：
1. 第一次 SpawnFromSave：创建 GameObject ①，Initialize 注册新 GUID，OverrideSaveId 换为存档 ID ✓
2. 第二次 SpawnFromSave：
   - 创建 GameObject ②（**未被销毁**）
   - Initialize 注册另一个新 GUID
   - OverrideSaveId 时先注销 GUID，再用存档 ID 注册 → `RegisterSaveable` 发现已存在 → 警告并忽略
   - GameObject ② 留在场景中，但 `LoadState` 阶段找不到它（_saveables 里没有它）

**触发条件**：存档文件被外部修改，或上游 `Save` 时出现重复 SaveId（理论上 GUID 不会重复，但要防御）

**修复建议**：
- `SpawnFromSave` 开头检查：`if (SaveManager.Instance.HasSaveable(entry.saveId)) { Debug.LogWarning(...); return; }`
- 或在 `SaveManager.Load` 阶段 1.5 之前对 `root.modules` 按 saveId 去重
- SaveManager 需要新增 `HasSaveable(string id)` 公开方法

---

### R4: OverrideSaveId 的注销-注册非原子【低风险】

**位置**：`UnitController.OverrideSaveId`

**现象**：
```csharp
public void OverrideSaveId(string id)
{
    if (!string.IsNullOrEmpty(SaveId))
    {
        SaveManager.Instance.UnregisterSaveable(this);  // ← 注销旧 ID
    }
    SaveId = id;
    SaveManager.Instance.RegisterSaveable(this);  // ← 注册新 ID
}
```

**影响**：
- 中间存在时间窗口，此时 `_saveables` 里没有该对象
- 当前是同步调用，时间窗口极小，但理论上存在
- 如果未来引入异步加载，风险会放大

**修复建议**：
- SaveManager 提供 `ChangeSaveId(string oldId, string newId, ISaveable saveable)` 方法
- 内部用字典的 Remove + Add 一次性完成（虽然 C# Dictionary 没有原子 rename，但封装后便于未来优化）

---

### R5: Singleton 自动创建空 RulerController 与 Prefab 实例分离【中风险】

**位置**：`RulerController.Awake`（重写后不调 `base.Awake()`）

**现象**：
1. 任何代码访问 `RulerController.Instance` 时，Singleton getter 可能自动创建空 GameObject
2. 空 GameObject 上的 RulerController.Awake 被调用：
   - `_instance == null` → 走 `_instance = this` 分支
   - `_isRealSingleton = true`
   - `DontDestroyOnLoad`
   - `TryLoadRulerData` → `rulerData == null`（空对象没有 SerializeField 引用）
3. 真正的 Prefab 实例化时：
   - RulerController.Awake 被调用
   - `_instance != null`（空对象）→ 走 `Destroy(this)` 分支
   - `_isRealSingleton` 保持 false
4. **结果：全局 `RulerController.Instance` 永远指向空对象，不是 Prefab 实例**

**影响**：
- `RulerController.Instance.transform` 不是君主的位置
- `RulerController.Instance.RulerData` 在 `OnUnitDataLoaded` 事件后才填充
- 架构上的隐式分离，后续开发容易引入新 bug（这是 Bug①③ 的共同根因）
- 与修复记录末尾的"架构建议"一致：**应从 Prefab 上移除 RulerController**

**触发条件**：任何代码在 Prefab 实例化前访问 `RulerController.Instance`
- 当前 `GameBootstrap.InitializeScene` 读档分支第一行 `RulerController.Instance.DestroyAllSceneMonarchs()` 就会触发

**修复建议**（与修复记录末尾一致，但更紧迫）：
- 把 `RulerController` 从 `Human_Player_Ruler.prefab` 上移除
- 在场景中创建独立的 `[Managers]` GameObject 挂载 `RulerController`
- 这样 Awake 时 `_instance` 必为空，不会触发自动创建副作用
- Prefab 实例化不再涉及 RulerController，所有 4 个 Bug 都不会发生
- **当前代码已防御各种情况，但隐式分离仍在；建议尽快重构**

---

### R6: SaveManager._saveables / UnitRegistry._aliveUnits 跨场景不清理【中风险】

**位置**：`Singleton<T>` 跨场景 `DontDestroyOnLoad`

**现象**：
- `SaveManager` 和 `UnitRegistry` 是 DontDestroyOnLoad 的 Singleton
- 场景中的 UnitController 不带 DontDestroyOnLoad，场景切换时被 Unity 销毁
- 但 `_saveables` 和 `_aliveUnits` 仍持有这些已销毁的引用（C# 引用不会自动消失）

**影响**：
- 内存泄漏：字典/HashSet 越来越大
- 查询返回已销毁对象：`UnitRegistry.GetEnemies()` 等返回 null 引用（Unity 重载了 == 但调用方未必 null check）
- 读档后旧场景单位的"幽灵引用"可能干扰新场景逻辑
- `SaveManager.Save` 里有 `if (saveable is MonoBehaviour mb && mb == null) continue;` 防御，但只是跳过，不清理

**触发条件**：玩家从主菜单读档进入游戏 → 退出到主菜单 → 再读档进入游戏（场景重载）

**修复建议**：
- 在 `GameBootstrap.Awake` 或场景切换时调用 `UnitRegistry.Clear()`
- SaveManager 新增 `CleanupDestroyedSaveables()` 方法：
  ```csharp
  public void CleanupDestroyedSaveables()
  {
      var toRemove = new List<string>();
      foreach (var kvp in _saveables)
      {
          if (kvp.Value is MonoBehaviour mb && mb == null)
              toRemove.Add(kvp.Key);
      }
      foreach (var id in toRemove) _saveables.Remove(id);
  }
  ```
- 在 `GameBootstrap.Awake` 顶部调用一次

## 三、修复优先级

| 优先级 | 风险 | 修复成本 | 触发概率 | 建议 |
|--------|------|----------|----------|------|
| **P0** | R2 场景预置单位双份 | 低 | 高（开发期常见） | 立即修复：扩展清理范围为所有 UnitController |
| **P0** | R5 RulerController 分离 | 中 | 已发生（Bug①③） | 按"架构建议"从 Prefab 移除，挂独立 GameObject |
| **P1** | R3 存档重复 SaveId | 低 | 低（但要防御） | SpawnFromSave 加去重检查 |
| **P1** | R6 跨场景不清理 | 中 | 中（多次场景切换后） | 场景切换时清理引用 |
| **P2** | R1 Singleton 自动创建 | 高 | 隐式存在 | 整体重构 Instance getter |
| **P2** | R4 OverrideSaveId 非原子 | 低 | 极低 | 提供 ChangeSaveId 方法 |

## 四、数据传递链稳定性建议（长期）

### 1. 入口统一
- 所有场景切换都经过 `GameBootstrap`，避免业务代码直接 `SceneManager.LoadScene`
- 跨场景参数（`IsLoadingSave`、`SaveSlotToLoad`、`NewGameConfig`）只允许 `GameBootstrap` 持有

### 2. 清理统一
- 场景切换前必须清理 `_saveables` 和 `_aliveUnits` 中的已销毁引用
- 提供 `SaveManager.CleanupDestroyedSaveables()` 和 `UnitRegistry.Clear()` 作为标准清理入口
- `GameBootstrap.Awake` 顶部强制调用一次

### 3. ID 唯一性保证
- `Save` 时检查 SaveId 唯一性（GUID 理论上唯一，但要防御）
- `Load` 时检查存档内 SaveId 唯一性，重复时保留第一个、警告其余
- `SpawnFromSave` 前检查是否已存在该 SaveId 的对象

### 4. 去重防御
- 所有"从存档恢复"的 Spawner 在创建实例前都检查重复
- `RulerController.SpawnMonarch` 的 4 步防御是范例，其他 Spawner 应模仿

### 5. 架构分离
- 管理器（Singleton + DontDestroyOnLoad）与运行时单位（Prefab 实例）严格分离
- **永远不要把 Singleton Manager 挂在会被 Instantiate 的 Prefab 上**
- 这是 Bug①②③④ 的共同根因，R5 是这个根因的延续

### 6. 数据传递路径文档化
- 每个 ISaveable 模块的 SaveState/LoadState 应有明确的"数据所有权"文档
- 避免多个模块修改同一份数据（如 RulerController 和 UnitController 都涉及君主属性）
- 当前架构里：君主的国家资源在 RulerController，君主单位属性在 UnitController，这是对的

## 五、与君主系统 Bug 的关系

| Bug | 对应风险 | 说明 |
|-----|----------|------|
| Bug① Destroy(gameObject) 销毁 Prefab | R5 | RulerController 挂在 Prefab 上的根因 |
| Bug② 新建后两个君主 | R5 + 防御不足 | 修复后引入 4 步防御，但根因仍在 |
| Bug③ 读档后两个君主 | R2 + R5 | DestroyAllSceneMonarchs 只清理君主，非君主单位未清理 |
| Bug④ UnregisterSaveable 空值异常 | R6 衍生 | 未初始化单位的 SaveId=null，是跨场景不清理的副作用 |

**结论**：当前 4 个 Bug 都已修复，但根因（R5 + R2 + R6）未消除。建议按 P0 优先级修复 R2 和 R5，从架构上消除隐患。
