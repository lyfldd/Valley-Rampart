# 2.4 销毁系统设计

> 对称于 2.3 加载系统设计。LoadManager 管加载时序，TeardownManager 管销毁时序。
> 核心难点：**销毁顺序**——依赖方先销毁，被依赖方后销毁，否则销毁过程中访问已销毁对象会崩。

---

## 一、设计目标

1. **统一销毁出口**：消除 PausePanel / GameOverPanel / GameBootstrap 各自写清理逻辑的不一致
2. **保证销毁顺序**：按依赖关系反拓扑序销毁，避免"销毁 A 时 A 还要访问已销毁的 B"
3. **主动销毁代替被动清理**：替代当前的 `CleanupDestroyedSaveables` / `UnitRegistry.Clear` 事后扫描兜底
4. **管理 DontDestroyOnLoad 对象**：哪些跨场景存活、哪些该销毁，集中决策

---

## 二、当前痛点

销毁逻辑散落在至少 6 处，无统一协调：

| 位置 | 做什么 | 问题 |
|------|--------|------|
| `UnitController.Die()` | UnregisterSaveable + Publish UnitDiedEvent + Unregister + Destroy | 单个单位死亡，正确但局部 |
| `RulerController.DestroyAllSceneUnits()` | 遍历场景单位反注册+销毁 | 只管 UnitController，不管 monarchUnit 引用清理的时序 |
| `RulerController.DestroyMonarchForMenuReturn()` | 反注册君主 + Destroy | 调用点靠 PausePanel 自觉，GameOverPanel 漏调 |
| `PausePanel.OnQuitClicked()` | Save + DestroyMonarch + DisableInput + LoadScene | 完整但独此一份 |
| `GameOverPanel.OnBackToMenuClicked()` | 只 LoadScene | **缺 DestroyMonarch + 缺 DisableInput**（Bug 6 根源） |
| `GameBootstrap.Awake()` | UnitRegistry.Clear + CleanupDestroyedSaveables | 事后兜底，说明销毁时机不可控 |

**根本问题**：返回主菜单有两条路径，做的事不一样；新增第三条路径还得再抄一遍，漏一个就出 bug。

---

## 三、销毁顺序分析（核心）

### 3.1 依赖关系矩阵

**规则**：A 销毁时需要访问 B → B 必须在 A 之后销毁（B 后销毁）。

| 依赖方 A | 销毁时要访问 | 被依赖 B | 顺序约束 |
|---------|-------------|---------|---------|
| UI Panel | EventBus.Unsubscribe | EventBus | Panel 先，EventBus 后 |
| UI Panel | SaveManager / RulerController / InputManager | 各 Manager | Panel 先，Manager 后 |
| UnitController | SaveManager.UnregisterSaveable | SaveManager | Unit 先，SaveManager 后 |
| UnitController | UnitRegistry.Unregister | UnitRegistry | Unit 先，UnitRegistry 后 |
| UnitController | EventBus.Publish(UnitDiedEvent) | EventBus | Unit 先，EventBus 后 |
| RulerController | EventBus.Unsubscribe | EventBus | Ruler 先，EventBus 后 |
| RulerController | SaveManager.UnregisterSaveable | SaveManager | Ruler 先，SaveManager 后 |
| TimeManager | EventBus.Unsubscribe | EventBus | Time 先，EventBus 后 |
| SaveManager | EventBus.Unsubscribe | EventBus | SaveManager 先，EventBus 后 |
| EventBus.Clear | 所有订阅者已 Unsubscribe | 全体订阅者 | **EventBus 最后** |

### 3.2 反拓扑销毁顺序

从依赖方到被依赖方，**依赖方先销毁，被依赖方后销毁**：

```
 ①  InputManager.DisableInput()          ← 最先：停输入，防销毁中误触事件
 ②  UI Panels                            ← 反订阅 GameStateChangedEvent 等 + 销毁面板
 ③  UnitController 们（场景内单位）        ← UnregisterSaveable + UnitRegistry.Unregister + Destroy
 ④  RulerController.monarchUnit 引用清理   ← 引用置 null（单位已在 ③ 销毁或随 LoadScene 销毁）
 ⑤  RulerController 反订阅                ← Unsubscribe<UnitDiedEvent> + UnregisterSaveable
 ⑥  TimeManager 反订阅                    ← Unsubscribe + UnregisterSaveable
 ⑦  DifficultyManager 反订阅              ← Unsubscribe + UnregisterSaveable
 ⑧  WorldSystem 反注册                    ← UnregisterSaveable
 ⑨  SaveManager 反订阅                    ← Unsubscribe<TimeDayChangedEvent> + <GameStateChangedEvent>
 ⑩  UnitRegistry.Clear()                 ← 清空单位注册表
 ⑪  SaveManager._saveables 清理           ← CleanupDestroyedSaveables 或直接 Clear
 ⑫  EventBus.Clear()                     ← 最后：所有订阅者已 Unsubscribe，安全清空
 ⑬  (仅退出时) 销毁 Singleton GameObject   ← 进程退出自然清理，或显式 Destroy
```

### 3.3 三个关键陷阱

**陷阱 1：`Destroy()` 是延迟的**

`Destroy(obj)` 标记对象待销毁，实际销毁在帧末。同一帧内 `Destroy(A)` 后，A 的 C# 引用还在（`A != null` 为 true），但 `A.gameObject == null`（Unity fake null）。

**后果**：不能靠"先 Destroy(A) 再访问 A 的字段"判断 A 是否已销毁，必须用 Unity null check（`A == null` 或 `A.gameObject == null`）。

**对策**：销毁流程里**先反注册（UnregisterSaveable / Unregister），再 Destroy**。反注册是纯 C# 字典操作，立即生效；Destroy 延迟到帧末不影响反注册。

---

**陷阱 2：`OnDestroy` 时序不可靠，不能做关键清理**

`Destroy(gameObject)` 后，该 GameObject 上所有组件的 `OnDestroy` 在帧末被调用。但帧末时**其他对象可能已经销毁**——比如 EventBus 已 Clear，此时 `OnDestroy` 里调 `EventBus.Unsubscribe` 会操作已清空的字典。

**后果**：把 Unsubscribe 放在 OnDestroy 里不安全。当前各 Manager 的 OnDestroy 反订阅在"返回主菜单"时不触发（Singleton 不销毁），在"退出游戏"时才触发，但那时顺序不可控。

**对策**：TeardownManager **显式调用反订阅**，不依赖 OnDestroy。OnDestroy 只做"TeardownManager 没覆盖到的兜底"。

---

**陷阱 3：DontDestroyOnLoad 对象不随 LoadScene 销毁**

当前 DontDestroyOnLoad 清单（全部是 Singleton）：

| 对象 | 跨场景存活 | 返回主菜单时该不该销毁 |
|------|-----------|---------------------|
| GameStateManager | 是 | 否（Core，保留） |
| InputManager | 是 | 否（Core，保留） |
| SaveManager | 是 | 否（Core，保留） |
| LoadManager | 是 | 否（Core，保留） |
| WorldSystem | 是 | 否（业务门面，保留，状态由 InitializeWorld 覆盖） |
| WorldManager | 是 | 否（同上） |
| TimeManager | 是 | 否（同上） |
| DifficultyManager | 是 | 否（同上） |
| UnitRegistry | 是 | 否（Core，保留，但要 Clear） |
| UnitFactory | 是 | 否（Core，保留） |
| RulerController | 是 | 否（业务，保留，状态由 InitializeWorld 覆盖） |
| **君主单位（UnitController）** | **否** | **LoadScene 自动销毁**（UnitFactory.SpawnUnit 没设 DontDestroyOnLoad） |

**关键发现**：拆分 RulerController 后，君主单位已经不是 DontDestroyOnLoad 了。`LoadScene` 会自动销毁它。但 `UnitController` 没有 `OnDestroy` 反注册，所以 `SaveManager._saveables` 和 `UnitRegistry._aliveUnits` 里会残留已销毁引用，需要事后清理。

**对策**：TeardownManager 在 LoadScene 前主动反注册 + 销毁君主单位，不靠 LoadScene 被动销毁 + 事后扫描。

## 四、TeardownManager 设计

### 4.1 职责定位

```
LoadManager（加载）  ←→  TeardownManager（销毁）
   Booting                    TeardownScene      （读档/新游戏前）
   StaticConfigs              TeardownForMenu    （返回主菜单）
   WorldInit / SaveRestore    TeardownForQuit    （退出游戏）
   Playing
```

- LoadManager 管构建，TeardownManager 管拆除
- TeardownManager 是 Singleton，DontDestroyOnLoad，自身最后销毁
- **不接管业务逻辑**（不调 Save、不 SetState），只管"按顺序反注册 + 销毁 + 清空注册表"

### 4.2 三级 Teardown API

```csharp
public class TeardownManager : Singleton<TeardownManager>
{
    // Level 1：场景级清理（读档前 / 新游戏前）
    // 清理场景内运行时对象，不动 Core Manager，不动业务 Manager 订阅
    public void TeardownScene();

    // Level 2：返回主菜单（场景级 + 业务重置）
    // TeardownScene + DisableInput + 业务 Manager 反订阅 + LoadScene
    public void TeardownForReturnToMenu(bool saveBeforeTeardown = true);

    // Level 3：退出游戏（全部清理）
    // 反订阅所有 + 清空注册表 + EventBus.Clear + 重置 static
    public void TeardownForQuit();
}
```

### 4.3 销毁阶段定义

```csharp
public enum TeardownPhase
{
    InputStopped,      // 输入已停
    UnitsDestroyed,    // 场景单位已销毁
    ManagerReset,      // 业务 Manager 状态已重置（仅重置，不反订阅不反注册）
    RegistriesCleared, // 注册表已清空
    EventBusCleared,   // 事件总线已清空（仅退出时）
    Done
}
```

## 五、三种销毁场景流程

### 5.1 Level 1：TeardownScene（读档前 / 新游戏前）

**调用方**：`GameBootstrap.ContinueFromSave()` 开头（替代当前的 `DestroyAllSceneUnits`）

```
TeardownScene()
  ├── ① 遍历 FindObjectsOfType<UnitController>()
  │     对每个 unit：
  │       SaveManager.UnregisterSaveable(unit)   ← 先反注册（纯字典操作，立即生效）
  │       UnitRegistry.Unregister(unit)
  │       Destroy(unit.gameObject)               ← 后销毁（延迟到帧末）
  ├── ② RulerController.monarchUnit = null       ← 清除君主引用
  └── ③ SaveManager.CleanupDestroyedSaveables()  ← 兜底（防遗漏）
```

**不动**：Core Manager、业务 Manager 的事件订阅、EventBus

**和当前 DestroyAllSceneUnits 的区别**：统一入口，不散落在 RulerController 里；顺序由 TeardownManager 保证。

---

### 5.2 Level 2：TeardownForReturnToMenu（返回主菜单）

**调用方**：`PausePanel.OnQuitClicked` 和 `GameOverPanel.OnBackToMenuClicked` 都调它

#### 保留 vs 销毁 vs 重置决策表

**关键事实**：所有 Manager 都挂在 MainMenuScene 的独立 GameObject 上（DontDestroyOnLoad），返回主菜单时它们**全都活着**。真正需要"销毁"的只有场景内运行时对象。

| 对象 | 返回主菜单时 | 原因 |
|------|------------|------|
| Core Manager（GameState/Input/Save/Load/UnitRegistry/UnitFactory） | **保留，状态不动** | 主菜单也要用。UnitRegistry 只 Clear 列表 |
| 业务 Manager（WorldSystem/WorldManager/Time/Difficulty/Ruler） | **保留，重置运行时状态** | Manager 不销毁，下次 InitializeWorld 覆盖。但状态要清空防残留 |
| 场景内 UnitController | **主动反注册 + 销毁** | 运行时对象，不该带到主菜单 |
| 君主单位 | **主动反注册 + 销毁** | 同上 |
| UI Panels | **LoadScene 自动销毁** | 场景内对象 |
| EventBus | **不 Clear** | Manager 都保留，订阅也保留，Clear 了反而要重新 Subscribe |
| SaveManager._saveables | **不清空** | 业务 Manager 的 ISaveable 注册保留，下次存档还要用 |

**核心洞察**：业务 Manager 只需"重置状态"，**不需要反订阅、不需要反注册 ISaveable**——因为 Manager 都保留（不销毁），订阅和注册都要继续用。反订阅 + 反注册 + EventBus.Clear 只在退出游戏（Level 3）才做。

#### 流程

```
TeardownForReturnToMenu(saveBeforeTeardown)
  ├── ① InputManager.DisableInput()              ← 最先：停输入
  ├── ② if (saveBeforeTeardown)
  │     SaveManager.Save(currentSlotId)           ← 保存（Pause 要存，GameOver 不存）
  ├── ③ TeardownScene()                           ← 复用 Level 1（反注册+销毁场景单位）
  ├── ④ RulerController.DestroyMonarchForMenuReturn()  ← 销毁君主单位（反注册+Destroy）
  ├── ⑤ 业务 Manager.ResetState()                 ← 只重置运行时状态，不反订阅不反注册
  │     TimeManager.ResetState()                  CurrentDay=1, 停止推进
  │     DifficultyManager.ResetState()            CurrentFactor=初始值
  │     RulerController.ResetState()              monarchUnit=null, 资源归零, RulerName="无名君主"
  │     (WorldSystem/WorldManager 由 ApplyConfig 覆盖，可不专门 Reset)
  ├── ⑥ UnitRegistry.Clear()                      ← 清空单位注册表
  ├── ⑦ SaveManager.CleanupDestroyedSaveables()   ← 清 ③④ 销毁的单位残留
  └── ⑧ SceneManager.LoadScene(mainMenuSceneName)
```

#### 为什么业务 Manager 不反订阅/不反注册

- **不反订阅**：Manager 都保留（DontDestroyOnLoad），下次进 GameScene 走 InitializeWorld，不重复订阅（Singleton 不重新 Awake）。订阅保留不影响
- **不反注册 ISaveable**：Manager 保留，它的 ISaveable 注册也要保留。下次存档还要保存 RulerController/TimeManager 的状态。反注册了反而下次存档丢数据
- **EventBus 不 Clear**：Core Manager 的订阅（如 SaveManager 订阅 GameStateChangedEvent）主菜单也要用。Clear 了 Core Manager 不会重新 Subscribe（不重走 Start）

---

### 5.3 Level 3：TeardownForQuit（退出游戏）

**调用方**：`Singleton<T>.OnApplicationQuit` 或 `TeardownManager.OnApplicationQuit`

```
TeardownForQuit()
  ├── ① InputManager.DisableInput()
  ├── ② TeardownScene()                           ← 清理场景对象
  ├── ③ 各 Manager 的 OnDestroy 由 Unity 自动调    ← 反订阅（Unsubscribe 对已清空字典是 no-op，安全）
  ├── ④ EventBus.Clear()                          ← 清空事件总线（Unsubscribe 已是 no-op，顺序不敏感）
  └── ⑤ 重置 Singleton statics
        _instance = null
        _isQuitting = true
```

**注意**：进程退出时 Unity 自动回收内存。Level 3 不需要显式调 Manager 反订阅——Unity 会调各 Manager 的 OnDestroy（里面有 Unsubscribe）。关键是 `EventBus.Clear()` 要在 OnDestroy 触发前或后都行：`Unsubscribe` 内部有 `if (!_subscribers.TryGetValue(...)) return` 防护，对已清空的字典是 no-op，不会报错。所以退出时顺序不敏感，Level 3 很轻量。

## 六、与现有系统集成

### 6.1 IResettable 接口（可选）

```csharp
public interface IResettable
{
    void ResetState();  // 重置运行时状态（不反订阅、不反注册 ISaveable）
}
```

让需要被 TeardownManager 重置的业务 Manager 实现此接口。TeardownManager.Level 2 按顺序调用。

**但当前 Manager 数量有限（约 10 个），可以不引入接口，直接在 TeardownManager 里硬编码调用顺序**。简单直接，看得见顺序。等 Manager 超过 15 个再考虑接口+注册表。

### 6.2 调用点改造

| 现有调用点 | 改造为 |
|-----------|--------|
| `GameBootstrap.ContinueFromSave` 里的 `DestroyAllSceneUnits` | `TeardownManager.Instance.TeardownScene()` |
| `PausePanel.OnQuitClicked` 全部逻辑 | `TeardownManager.Instance.TeardownForReturnToMenu(saveBeforeTeardown: true)` |
| `GameOverPanel.OnBackToMenuClicked` 全部逻辑 | `TeardownManager.Instance.TeardownForReturnToMenu(saveBeforeTeardown: false)` |
| `RulerController.DestroyAllSceneUnits` | 删除（职责移到 TeardownManager） |
| `RulerController.DestroyMonarchForMenuReturn` | 保留，由 TeardownManager 内部调用 |
| `Singleton.OnApplicationQuit` | 调 `TeardownManager.Instance.TeardownForQuit()` |

### 6.3 与 LoadManager 的配合

```
GameScene 启动流程（读档模式）：
  GameBootstrap.Awake
    → TeardownManager.TeardownScene()     ← 先清理旧场景（如果有）
    → LoadManager.LoadStaticConfigs()      ← 再加载
  GameBootstrap.OnConfigsLoaded
    → LoadManager.LoadSave()               ← 存档恢复
    → RulerController.BindExistingMonarch()
```

TeardownManager 在 LoadManager 之前跑——先拆再建。

## 八、待确认点

1. **业务 Manager 的 ResetState 要不要彻底重置所有字段？**
   - 比如 TimeManager.ResetState 要不要把 CurrentDay 重置为 1、CurrentSeason 重置为 Spring？
   - 倾向：**要**。返回主菜单后再开新游戏，如果 InitializeWorld 没覆盖某个字段，残留旧值会出问题。ResetState 时彻底重置最安全。
   - 但要确认 InitializeWorld 是否覆盖了所有字段——如果全覆盖，ResetState 只做"停止运行"也行（如 TimeManager 停止推进，值由 InitializeWorld 覆盖）。

2. **SaveManager 的 GameStateChangedEvent 订阅要不要在 Level 2 保留？**
   - 倾向：保留。主菜单状态切换也要 SaveManager 响应（比如标记存档结束）。
   - 所以 Level 2 只反订阅 TimeDayChangedEvent，保留 GameStateChangedEvent。

3. **是否引入 ITeardownable 接口？**
   - 倾向：暂不。硬编码顺序，Manager 数量少时更清晰。等超过 15 个再重构。

## 九、设计原则总结

1. **对称**：LoadManager 管加载，TeardownManager 管销毁，职责对称
2. **顺序优先**：销毁顺序是核心，反拓扑序（依赖方先，被依赖方后）
3. **主动优于被动**：显式反注册 + 销毁，不靠 LoadScene 被动销毁 + 事后扫描
4. **不依赖 OnDestroy**：OnDestroy 时序不可控，TeardownManager 显式调用
5. **简单优于灵活**：硬编码顺序，不引入接口+注册表（Manager 数量未到那个量级）
6. **Core 与业务分离**：Core Manager 跨场景保留，业务 Manager Teardown 重置
