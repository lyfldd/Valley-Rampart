# 📦 加载系统设计（LoadManager 统一加载）

> 2026-07-23 阿铁拍板，主理人游承峰汇编
> 设计哲学：LoadManager 既是调度者（管加载时序）又是门面（统一取资源入口），"打包好之后其他系统去拿"

## 📝 设计动机

现有加载散在 4 处（UnitDataManager/UnitFactory/SaveManager/GameBootstrap），时序靠 Singleton 隐式创建撑着，配置和存档混在 GameBootstrap 一个方法里。需要一个统一系统接管所有加载，分清"一次性配置加载""新建/读档都要的""存档恢复""运行时动态"四类。

---

## 一、设计哲学

| 角色 | 职责 | 谁来当 |
|------|------|--------|
| **调度者** | 管加载时序、阶段化调度、加载完成通知 | LoadManager |
| **门面** | 对外统一取资源接口，其他系统只认 LoadManager | LoadManager |
| **子加载器** | 各自管自己的缓存和加载细节 | UnitDataManager / UnitFactory / SaveManager |
| **消费者** | 通过 LoadManager 取资源，不直接碰子加载器 | 其他所有系统 |

**核心原则**：
1. LoadManager 不自己缓存资源——各子加载器（UnitDataManager/UnitFactory）保持自己的缓存
2. LoadManager 委托转发——其他系统调 `LoadManager.Instance.GetUnitData()`，LoadManager 转发给 UnitDataManager
3. 加载时序由 LoadManager 管各子加载器何时加载，不靠 Awake 隐式触发
4. 阶段化加载——每阶段有明确入口和完成标志，其他系统订阅加载完成事件再取资源

## 三、加载阶段定义

```
Phase 0: Booting（引擎初始化）
  │  LoadManager.Awake → 校验关键 Singleton 都在场景里（不靠隐式创建）
  │  GameStateManager.SetState(Booting)
  ▼
Phase 1: StaticConfigs（静态配置加载，新建/读档都要）
  │  LoadManager.LoadStaticConfigs()
  │    ├─ UnitDataManager.LoadAll()         ← Resources/UnitData/ SO
  │    ├─ UnitFactory.PreloadAll()          ← Resources/UnitPrefabs/ Prefab
  │    └─ WorldSystem.LoadWorldConfig()     ← Resources/World/WorldConfig SO
  │  发布 ConfigsLoadedEvent
  │  GameStateManager.SetState(Loading → Ready)
  ▼
Phase 2a: WorldInit（仅新建游戏）
  │  LoadManager.InitializeNewGame(NewGameConfig)
  │    └─ WorldSystem.InitializeWorld(NewGameConfig)
  │         ├─ WorldManager.ApplyConfig(mapSeed, difficulty)
  │         ├─ DifficultyManager.Initialize(difficulty)
  │         ├─ TimeManager 配置（从 WorldConfig 读）
  │         └─ RulerController.SetRulerName + SpawnMonarch
  ▼
Phase 2b: SaveRestore（仅读档）
  │  LoadManager.LoadSave(slotId)
  │    └─ SaveManager.Load(slotId)
  │         ├─ Global 阶段：WorldSystem/TimeManager/DifficultyManager/WorldManager.LoadState
  │         ├─ Scene 重建：UnitFactory.SpawnFromSave（ISaveableSpawner）
  │         └─ Scene 恢复：单位 LoadState
  ▼
Phase 3: Playing（运行时，按需动态加载）
  │  LoadManager.LoadAsset<T>(path)  ← 建筑/敌人/VFX 按需加载
  │  GameStateManager.SetState(Playing)
```

**关键**：Phase 1 是新建和读档**都要走**的，配置加载只做一次。Phase 2 才分叉。

## 四、LoadManager 完整设计

**文件**：`Assets/_Game/Systems/Loading/LoadManager.cs`

```csharp
using UnityEngine;

/// <summary>
/// 加载阶段。决定当前加载进度，其他系统可查询后决定是否取资源。
/// </summary>
public enum LoadPhase
{
    Booting,         // 引擎初始化
    StaticConfigs,   // 静态配置加载中
    WorldInit,       // 新建游戏初始化中
    SaveRestore,     // 读档恢复中
    Playing          // 运行时，可按需动态加载
}

/// <summary>
/// 加载总管。统一加载入口 + 资源获取门面。
/// 职责：阶段化调度加载 + 委托转发资源请求。
/// 不自己缓存资源——各子加载器保持自己的缓存。
/// </summary>
[DefaultExecutionOrder(-110)]  // 比 GameBootstrap(-100) 更早
public class LoadManager : Singleton<LoadManager>
{
    // ===== 加载阶段状态 =====

    public LoadPhase CurrentPhase { get; private set; } = LoadPhase.Booting;

    // ===== 子加载器引用（场景里挂的 Singleton，启动后获取）=====

    private UnitDataManager _configLoader;
    private UnitFactory _prefabLoader;
    private SaveManager _saveLoader;
    private WorldSystem _worldSystem;

    protected override void Awake()
    {
        base.Awake();

        // 校验关键 Singleton（不靠隐式创建，要求场景里显式放置）
        _configLoader = FindObjectOfType<UnitDataManager>();
        _prefabLoader = FindObjectOfType<UnitFactory>();
        _saveLoader = SaveManager.Instance;
        _worldSystem = FindObjectOfType<WorldSystem>();

        if (_configLoader == null) Debug.LogError("[LoadManager] 场景中未找到 UnitDataManager！");
        if (_prefabLoader == null) Debug.LogError("[LoadManager] 场景中未找到 UnitFactory！");
        if (_worldSystem == null) Debug.LogError("[LoadManager] 场景中未找到 WorldSystem！");
    }

    // ===== 阶段1：静态配置加载（新建/读档都要）=====

    /// <summary>
    /// 加载所有静态配置（SO + Prefab）。新建和读档都调用一次。
    /// 由 GameBootstrap 在 Loading 状态时调。
    /// </summary>
    public void LoadStaticConfigs()
    {
        CurrentPhase = LoadPhase.StaticConfigs;
        Debug.Log("[LoadManager] 阶段1：加载静态配置...");

        // 1. SO 配置
        _configLoader.LoadAll();           // UnitData 等
        // 未来：_configLoader.LoadAll<BuildingData>("BuildingData");

        // 2. Prefab 预加载
        _prefabLoader.PreloadAll();        // UnitPrefabs
        // 未来：_prefabLoader.PreloadAll("BuildingPrefabs");

        // 3. WorldConfig（WorldSystem 自己加载，这里只确认）
        Debug.Log("[LoadManager] 静态配置加载完成");
        EventBus.Publish(new ConfigsLoadedEvent(true));
    }

    // ===== 阶段2a：新建游戏初始化 =====

    /// <summary>
    /// 新建游戏：世界初始化。由 GameBootstrap 在 Ready 状态、新建模式时调。
    /// </summary>
    public void InitializeNewGame(NewGameConfig config)
    {
        CurrentPhase = LoadPhase.WorldInit;
        Debug.Log("[LoadManager] 阶段2a：新建游戏初始化...");

        _worldSystem.InitializeWorld(config);

        Debug.Log("[LoadManager] 新建游戏初始化完成");
        EnterPlaying();
    }

    // ===== 阶段2b：读档恢复 =====

    /// <summary>
    /// 读档：存档恢复。由 GameBootstrap 在 Ready 状态、读档模式时调。
    /// </summary>
    public void LoadSave(string slotId)
    {
        CurrentPhase = LoadPhase.SaveRestore;
        Debug.Log($"[LoadManager] 阶段2b：读档恢复 slot={slotId}...");

        _saveLoader.Load(slotId);
        // WorldSystem.LoadState 会通过 ISaveable 自动恢复
        // 并调 ApplyTimeConfig + DifficultyManager.SyncConfigFromWorld

        Debug.Log("[LoadManager] 读档恢复完成");
        EnterPlaying();
    }

    // ===== 阶段3：进入运行时 =====

    private void EnterPlaying()
    {
        CurrentPhase = LoadPhase.Playing;
        GameStateManager.Instance.SetState(GameState.Playing);
        Debug.Log("[LoadManager] 进入运行时，可按需动态加载");
    }

    // ===== 运行时动态加载（未来扩展）=====

    /// <summary>
    /// 运行时按需加载资源（建筑/敌人/VFX Prefab 等）。
    /// </summary>
    public T LoadAsset<T>(string path) where T : Object
    {
        T asset = Resources.Load<T>(path);
        if (asset == null)
        {
            Debug.LogError($"[LoadManager] 找不到资源: {path}");
        }
        return asset;
    }

    // ===== 门面接口：资源获取（委托转发）=====

    /// <summary>获取单位配置（委托 UnitDataManager）。</summary>
    public UnitData GetUnitData(Faction faction, Occupation occupation)
    {
        return _configLoader.GetData(faction, occupation);
    }

    /// <summary>获取所有单位配置（委托 UnitDataManager）。</summary>
    public System.Collections.Generic.IEnumerable<UnitData> GetAllUnitData()
    {
        return _configLoader.GetAllData();
    }

    /// <summary>获取单位 Prefab（委托 UnitFactory）。</summary>
    public GameObject GetPrefab(string key)
    {
        return _prefabLoader.GetPrefab(key);
    }

    /// <summary>创建单位实例（委托 UnitFactory）。</summary>
    public GameObject SpawnUnit(UnitData data, Vector2 position)
    {
        return _prefabLoader.SpawnUnit(data, position);
    }

    /// <summary>世界配置（委托 WorldSystem）。</summary>
    public WorldConfig WorldConfig => _worldSystem.Config;

    /// <summary>当前难度系数（委托 DifficultyManager）。</summary>
    public float CurrentDifficultyFactor => DifficultyManager.Instance.CurrentFactor;

    /// <summary>存档是否存在（委托 SaveManager）。</summary>
    public bool HasSave(string slotId) => _saveLoader.HasSave(slotId);
}

/// <summary>静态配置加载完成事件。</summary>
public readonly struct ConfigsLoadedEvent
{
    public readonly bool IsSuccess;
    public ConfigsLoadedEvent(bool isSuccess) { IsSuccess = isSuccess; }
}
```

## 五、资源获取门面接口（消费者用法）

其他系统**只认 LoadManager**，不直接碰 UnitDataManager/UnitFactory：

```csharp
// ❌ 旧写法（直接碰子加载器，时序不可控）
UnitData data = UnitDataManager.Instance.GetData(faction, occupation);
GameObject prefab = UnitFactory.Instance.GetPrefab(key);

// ✅ 新写法（通过 LoadManager 门面，统一入口）
UnitData data = LoadManager.Instance.GetUnitData(faction, occupation);
GameObject prefab = LoadManager.Instance.GetPrefab(key);
WorldConfig config = LoadManager.Instance.WorldConfig;
float factor = LoadManager.Instance.CurrentDifficultyFactor;
```

**好处**：
1. 消费者不需要知道资源在哪个子加载器里
2. LoadManager 保证阶段1完成后才允许取（CurrentPhase == Playing 时资源一定就绪）
3. 未来加新的资源类型（BuildingData/EnemyData），只改 LoadManager 门面，消费者无感

## 六、和现有系统衔接（改造清单）

### 6.1 GameBootstrap 改造

**改动**：协调逻辑移到 LoadManager，GameBootstrap 只负责"触发"。

```csharp
// 改动后的 GameBootstrap
[DefaultExecutionOrder(-100)]
public class GameBootstrap : MonoBehaviour
{
    private bool _sceneInitialized;

    public static bool IsLoadingSave = false;
    public static string SaveSlotToLoad;
    public static NewGameConfig NewGameConfig;

    private void Awake()
    {
        Debug.Log("[GameBootstrap] 游戏启动");
        SaveManager.Instance.RegisterSpawner(UnitFactory.Instance);
        EventBus.Subscribe<ConfigsLoadedEvent>(OnConfigsLoaded);
    }

    private void Start()
    {
        GameStateManager.Instance.SetState(GameState.Loading);
        // 触发阶段1：LoadManager 加载静态配置
        LoadManager.Instance.LoadStaticConfigs();
    }

    private void OnDestroy()
    {
        EventBus.Unsubscribe<ConfigsLoadedEvent>(OnConfigsLoaded);
        IsLoadingSave = false;
        SaveSlotToLoad = null;
        NewGameConfig = null;
    }

    // 阶段1完成回调 → 进阶段2
    private void OnConfigsLoaded(ConfigsLoadedEvent evt)
    {
        if (!evt.IsSuccess || _sceneInitialized) return;
        _sceneInitialized = true;

        GameStateManager.Instance.SetState(GameState.Ready);

        if (IsLoadingSave)
        {
            // 阶段2b：读档
            LoadManager.Instance.LoadSave(SaveSlotToLoad);
            IsLoadingSave = false;
            SaveSlotToLoad = null;
        }
        else
        {
            // 阶段2a：新建游戏
            LoadManager.Instance.InitializeNewGame(NewGameConfig);

            // 新建后自动存档
            string slotId = NewGameConfig?.selectedSlotId ?? "slot_1";
            if (!SaveManager.Instance.HasSave(slotId))
            {
                SaveManager.Instance.Save(slotId);
            }
            NewGameConfig = null;
        }
    }
}
```

**关键变化**：
- 删除 `OnDataLoaded`（订阅 UnitDataLoadedEvent）→ 改订阅 `ConfigsLoadedEvent`
- 删除 `InitializeScene` 大方法 → 拆成 LoadManager 的阶段2a/2b
- 删除 `UnitDataManager.IsInitialized` 检查 → LoadManager 保证时序

## 七、加载时序图

### 7.1 新建游戏

```

  └─ RegisterSpawner(UnitFactory)
  └─ Subscribe(ConfigsLoadedEvent)

  └─ GameStateManager.SetState(Loading)
  └─ LoadManager.LoadStaticConfigs()           ← 阶段1
```

## 十二、Addressables 迁移路径（未来热更新）

阿铁明确未来想要热更新，所以 LoadManager 的设计必须预留 Addressables 迁移路径。**好消息：门面模式天然隔离了加载实现，迁移很方便。**

### 12.1 为什么能方便迁移

```
消费者代码（不关心底层是 Resources 还是 Addressables）
  │
  ▼
LoadManager 门面接口（GetUnitData / GetPrefab / LoadAsset<T>）
  │
  ▼
┌─────────────────┐     ┌─────────────────────┐
│ 现在实现          │     │ 迁移后实现            │
│ Resources.Load   │ →   │ Addressables.Load   │
│ （同步）          │     │ （异步）              │
└─────────────────┘     └─────────────────────┘
```

消费者只调 `LoadManager.Instance.GetXxx()`，不碰底层。迁移时**只改 LoadManager 内部实现**，消费者代码零改动。

### 12.2 迁移步骤（未来执行）

| 步骤 | 内容 | 改动范围 |
|------|------|---------|
| 1 | Unity Editor 里把 `Resources/` 下资源标记为 Addressable（勾选 + 设地址 key） | Editor 操作，无代码 |
| 2 | LoadManager 阶段1 预加载：`Resources.LoadAll` → `Addressables.LoadAssetsAsync` | 改 LoadManager 内部 |
| 3 | LoadManager 阶段3 动态加载：`Resources.Load` → `Addressables.LoadAssetAsync` | 改 LoadManager 内部 |
| 4 | 子加载器（UnitDataManager/UnitFactory）的加载方法改异步 | 改子加载器内部 |
| 5 | 消费者代码 | **零改动**（门面接口不变） |

### 12.3 现在就要做的设计预留

为了让未来迁移最平滑，LoadManager 现在就把"同步预加载"和"可异步动态加载"分开设计：

| 接口类型 | 现在实现 | 迁移时改动 | 消费者影响 |
|---------|---------|-----------|-----------|
| **预加载接口**（阶段1 LoadStaticConfigs） | 同步 `Resources.LoadAll` | 改协程 `IEnumerator` + `yield return` | 无（阶段1由 LoadManager 内部驱动，消费者只等 ConfigsLoadedEvent） |
| **缓存取资源接口**（GetUnitData/GetPrefab） | 同步取缓存 | **不变**（资源已预加载在缓存里，取是同步的） | **零改动** |
| **运行时动态加载接口**（LoadAsset<T>） | 同步 `Resources.Load` | 改 `Task<T>` 或 `IEnumerator` | 调用处加 `await` 或 `StartCoroutine` |

**关键洞察**：
- 预加载（阶段1）改异步，消费者无感——因为消费者是订阅 `ConfigsLoadedEvent` 才往下走，加载是同步还是异步不影响消费者
- 缓存取资源永远是同步的——资源已经在内存里，取就是查字典，不需要异步
- 只有"运行时动态加载"（LoadAsset<T>）迁移时需要消费者配合改异步

所以迁移时，**99% 的消费者代码零改动**，只有少数"运行时按需加载"的调用处需要加 `await`。

### 12.4 迁移决策树（什么时候迁）

```
现在（2026-07）→ Resources
  │
  ├─ 资源量 < 200 个？ → 继续 Resources
  ├─ 需要热更新/远程下载？ → 迁移 Addressables
  ├─ 包体 > 500MB 需要瘦身？ → 迁移 Addressables
  └─ 团队规模扩大、多人协作资源？ → 迁移 Addressables
```

现阶段满足"资源量小 + 无远程需求"，用 Resources。未来任何一个条件触发，按 12.2 步骤迁移即可。
