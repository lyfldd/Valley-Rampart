## 5. ✏️ `TimeManager.cs`（修改）

**路径：** `Assets/_Game/Systems/Time/TimeManager.cs`

### 改动点

| 位置 | 改什么 |
|---|---|
| 类声明 | `: Singleton<TimeManager>` → `: Singleton<TimeManager>, ISaveable` |
| `Awake` 末尾 | 加 `SaveManager.Instance.RegisterSaveable(this);` |
| 类内 | 加 `public string SaveId => "TimeManager";` |

| 类内 | 加 `LoadState(SavePayload)` 方法 |
| 同文件末尾 | 加 `[Serializable] class TimeSaveData { ... }` |

### 删除/不需要的

- v1 文档里提到的 `RestoreState(int, float, Season, TimePhase)` 方法**不要加**——被 `LoadState` 替代了。

### 关键细节

- `LoadState` 里**先调 `SetSecondsPerDay` / `SetTotalDays` / `SetDaysPerSeason`**，再赋值 `CurrentDay` 等运行时字段。防止 `AdvanceDay` 误触发。
- `LoadState` 直接给私有字段赋值，**不调 `AdvanceDay`**，**不发事件**。否则会触发敌人刷新/季节变化逻辑。

### ⚠️ 注册时机陷阱

`TimeManager.Awake` 里调 `SaveManager.Instance` 会触发 `SaveManager` 单例创建。要保证 `SaveManager` 的 `Awake` 不依赖 `TimeManager`（目前 `SaveManager` 没有业务依赖，OK）。

执行顺序靠 `[DefaultExecutionOrder]`：
- `GameBootstrap` 已是 `-100`
- 建议 `SaveManager` 设 `-90`（在 GameBootstrap 之前，业务系统之后）
- `TimeManager` / `RulerController` 不用设，它们 `Awake` 访问 `SaveManager.Instance` 时会自动创建

## 6. ✏️ `RulerController.cs`（修改）

**路径：** `Assets/_Game/Systems/Ruler/RulerController.cs`

### 改动点

| 位置 | 改什么 |
|---|---|
| 类声明 | 加 `, ISaveable` |
| `Awake` 末尾 | 加 `SaveManager.Instance.RegisterSaveable(this);` |
| 类内 | 加 `public string SaveId => "RulerController";` |
| 类内 | 加 `public SaveLoadPhase LoadPhase => SaveLoadPhase.Global;` |
| 类内 | 加 `SaveState()` / `LoadState()` |
| 同文件末尾 | 加 `[Serializable] class RulerSaveData { int gold, stone, wood, food; }` |

### ⚠️ 重要逻辑改动

**`LoadState` 里不要调 `SpawnMonarch`。**

v1 文档里 `RulerController.RestoreResources` 之后会触发君主创建——v2 架构下这会导致状态不一致：资源数值恢复了，但君主 GameObject 还没生成（要等 `UnitFactory.SpawnFromSave` 在 Scene 阶段创建）。

v2 的正确流程：
1. Global 阶段：`RulerController.LoadState` 只恢复 `Gold` / `Stone` / `Wood` / `Food` 四个数值
2. Scene 阶段：`UnitFactory.SpawnFromSave` 创建君主 GameObject（君主在 `units` 列表里，`occupation == Ruler`）
3. Scene 阶段：`UnitController.LoadState` 恢复君主的 HP/位置等

### 现有逻辑保留

`SpawnMonarch()` 方法**保留**——新建游戏时（不是读档）由 `GameBootstrap.InitializeScene` 调用。只是 `LoadState` 里不再调它。

### 新建游戏 vs 读档的分叉

`GameBootstrap.InitializeScene` 里现在是：
```csharp
RulerController.Instance.SpawnMonarch();  // 新建游戏走这条
```

读档时应该跳过这行，让 `UnitFactory.SpawnFromSave` 接管。需要在 `GameBootstrap` 里加个分支判断（见 #9）。

## 7. ✏️ `UnitController.cs`（修改）

**路径：** `Assets/_Game/Systems/Unit/UnitController.cs`

这是改动最大的文件。

### 改动点

| 位置 | 改什么 |
|---|---|
| 类声明 | 加 `, ISaveable` |
| 字段 | 加 `public string SaveId { get; private set; }` |
| 类内 | 加 `public SaveLoadPhase LoadPhase => SaveLoadPhase.Scene;` |
| `Initialize(UnitData)` | 末尾加：分配 `SaveId` + `RegisterSaveable` |
| `Die()` | **开头**加 `SaveManager.Instance.UnregisterSaveable(this);` |
| 类内 | 加 `SaveState()` / `LoadState(SavePayload)` |
| 同文件末尾 | 加 `[Serializable] class UnitSaveData { ... }` |

### `Initialize` 改动细节

```csharp
public virtual void Initialize(UnitData data)
{
    Data = data;
    // ... 原有初始化逻辑 ...

    UnitRegistry.Instance.Register(this);
    EventBus.Publish(new UnitSpawnedEvent(this));

    // —— 新增：分配唯一 SaveId 并注册为可存档对象 ——
    SaveId = $"Unit_{data.faction}_{data.occupation}_{System.Guid.NewGuid():N}";
    SaveManager.Instance.RegisterSaveable(this);
}
```

### `Die` 改动细节（关键）

```csharp
protected virtual void Die()
{
    Debug.Log($"[UnitController] {Data?.faction}_{Data?.occupation} 死亡。");

    // —— 新增：先注销 ISaveable，再销毁对象 ——
    // 否则下次 SaveGame 会抓到已销毁实例，触发 MissingReferenceException
    SaveManager.Instance.UnregisterSaveable(this);

    EventBus.Publish(new UnitDiedEvent(this));
    UnitRegistry.Instance.Unregister(this);
    Destroy(gameObject);
}
```

**顺序很重要：** 注销必须在 `Destroy(gameObject)` 之前。`Destroy` 是延迟到帧末执行，但即使如此，注销要早于事件发布——因为 `UnitDiedEvent` 的订阅者可能触发存档（比如死亡时自动存档），那时对象还没注销就会出问题。

### ⚠️ 场景里手动放置的单位

`RulerController.FindExistingMonarch` 会找到场景里手动放的单位，调 `Initialize(rulerData)`。这条路径也会触发 `SaveId` 分配和注册——OK，没问题。

### ⚠️ 读档时的双重初始化

`UnitFactory.SpawnFromSave` 调 `SpawnUnit(config, pos)` → `Initialize(data)` → 注册 ISaveable（SaveId 是新 GUID）。然后 `SaveManager` 在 Scene 阶段分发 `LoadState`——按 SaveId 匹配。

**问题：** 读档时分配的 SaveId 是新 GUID，和存档里的 SaveId 不一样，`DistributePayloads` 按 SaveId 查不到对象！

**解决方案：** `UnitFactory.SpawnFromSave` 创建实例后，**手动把存档里的 SaveId 赋给新实例**，覆盖 `Initialize` 里分配的新 GUID。

```csharp
// UnitFactory.SpawnFromSave
public void SpawnFromSave(SavePayload payload)
{
    var data = JsonUtility.FromJson<UnitSaveData>(payload.json);
    // ... SpawnUnit ...

    var controller = go.GetComponent<UnitController>();
    controller.OverrideSaveId(payload.saveId);  // ← 新增方法
}
```

`UnitController` 加：
```csharp
public void OverrideSaveId(string id)
{
    if (string.IsNullOrEmpty(SaveId)) { SaveId = id; return; }
    // 已注册过，先注销再重新注册
    SaveManager.Instance.UnregisterSaveable(this);
    SaveId = id;
    SaveManager.Instance.RegisterSaveable(this);
}
```

**注意 `SpawnFromSave` 需要拿到存档里的 SaveId**——但 `SavePayload` 里没有 SaveId 字段（只有 typeName/json/version）。这需要 `ISaveableSpawner.SpawnFromSave` 的签名带上 SaveId，或者改用 `ModuleSaveEntry` 作为参数。

**推荐改法：** 把 `ISaveableSpawner.SpawnFromSave` 的参数从 `SavePayload` 改成 `ModuleSaveEntry`（包含 saveId）：

```csharp
public interface ISaveableSpawner
{
    string SaveIdPrefix { get; }
    void SpawnFromSave(ModuleSaveEntry entry);  // ← 改参数类型
}
```

`SaveManager.Load` 里调用时直接传 `entry`，spawner 拿到 `entry.saveId` 后传给新实例。**这个改动要同步更新 v2 文档第 4.3 节的代码示例。**

## 8. ✏️ `UnitFactory.cs`（修改）

**路径：** `Assets/_Game/Systems/Unit/UnitFactory.cs`

### 改动点

| 位置 | 改什么 |
|---|---|

| 类内 | 加 `public string SaveIdPrefix => "Unit_";` |

### `SpawnFromSave` 实现

```csharp
public void SpawnFromSave(ModuleSaveEntry entry)
{
    if (entry.typeName != typeof(UnitSaveData).AssemblyQualifiedName) return;

    var data = JsonUtility.FromJson<UnitSaveData>(entry.json);
    var faction = (Faction)data.faction;
    var occupation = (Occupation)data.occupation;

    UnitData config = UnitDataManager.Instance.GetData(faction, occupation);
    if (config == null)
    {
        Debug.LogError($"[UnitFactory] 找不到配置: {faction}_{occupation}，跳过。");
        return;
    }

    Vector2 pos = new Vector2(data.posX, data.posY);
    GameObject go = SpawnUnit(config, pos);  // 触发 Initialize → 注册 ISaveable（新 GUID）

    if (go != null)
    {
        var controller = go.GetComponent<UnitController>();
        controller.OverrideSaveId(entry.saveId);  // 覆盖为存档里的 SaveId
    }
}
```

### 现有 `PreloadAll` / `SpawnUnit` 保留

不动。`SpawnFromSave` 内部复用 `SpawnUnit`。

### 注册 Spawner

```csharp
private void Awake()
{
    Debug.Log("[GameBootstrap] 游戏启动");
    GameStateManager.Instance.SetState(GameState.Booting);
    EventBus.Subscribe<UnitDataLoadedEvent>(OnDataLoaded);

    // —— 新增：注册场景对象重建器 ——
    SaveManager.Instance.RegisterSpawner(UnitFactory.Instance);

```

### 新建/读档分叉

`InitializeScene` 现在是新建游戏的流程。读档时不应该 `SpawnMonarch`（君主由 `UnitFactory.SpawnFromSave` 创建），应该调 `SaveManager.Load`。

需要一个标志区分两种模式。用静态字段（场景切换间传参）：

```csharp
public static bool IsLoadingSave = false;
public static string SaveSlotToLoad;

private void InitializeScene()
{
    UnitFactory.Instance.PreloadAll();

    if (IsLoadingSave)
    {
        // 读档模式：不 SpawnMonarch，由 SaveManager 接管
        SaveManager.Instance.Load(SaveSlotToLoad);
        IsLoadingSave = false;
        SaveSlotToLoad = null;
    }
    else
    {
        // 新建模式：原有流程
        RulerController.Instance.SpawnMonarch();
    }

    var timeManager = TimeManager.Instance;
    Debug.Log($"[GameBootstrap] 时间系统就绪: 第{timeManager.CurrentDay}天 ...");

    InputManager.Instance.EnableInput();
    GameStateManager.Instance.SetState(GameState.Playing);
}
```

**注意：** 读档模式下 `TimeManager` 的 `Awake` 已经把默认值设好了（第 1 天早晨），然后 `SaveManager.Load` 的 Global 阶段会用存档值覆盖。`InputManager.EnableInput` 和 `SetState(Playing)` 在 Load 之后才执行，OK。

**⚠️ 执行顺序问题：** `SaveManager.Load` 是同步的，但 `UnitFactory.SpawnFromSave` 创建的单位的 `Initialize` 会触发 `UnitSpawnedEvent`——此时 `GameState` 还是 `Ready`，订阅者如果检查 `GameState == Playing` 会跳过。这是预期行为（读档期间不该响应单位生成事件），但如果某个系统依赖这个事件初始化，需要单独处理。

## 13. 风险点汇总

| 风险 | 影响 | 缓解 |
|---|---|---|
| `SaveManager.Instance` 在 `TimeManager.Awake` 时还没创建 | 注册失败 | Singleton 访问会自动创建，OK；但要保证 `SaveManager.Awake` 不反向依赖业务系统 |
| 读档时 `SaveId` 匹配不上 | 单位 `LoadState` 不被调用，状态丢失 | `UnitFactory.SpawnFromSave` 里 `OverrideSaveId` 把存档 SaveId 赋给新实例 |
| 读档时单位 `Initialize` 触发 `UnitSpawnedEvent` | 订阅者可能在 `GameState != Playing` 时响应 | 预期行为，订阅者应检查 GameState；或读档期间设一个 `IsLoading` 标志全局屏蔽 |
| `JsonUtility` 反序列化失败（存档格式变了） | 整个存档报废 | `LoadState` 里 try-catch，单模块失败不阻塞其他模块 |
| 场景里手动放的单位没经过 `Initialize` | `SaveId` 为 null，注册失败 | `UnitController.Awake` 里给 `SaveId` 一个临时值，或要求所有单位必经 `UnitFactory.SpawnUnit` 创建 |

## 14. 自动存档触发点（本次改造包含）

自动存档存到 `SaveManager.CurrentSlotId`（当前正在玩的槽位），不需要单独的 autosave 槽。

### 触发时机

| 时机 | 触发方式 | 实现 |
|---|---|---|
| **每 N 天** | `TimeManager` 发布 `TimeDayChangedEvent` | `SaveManager` 订阅，按 `AutoSaveIntervalDays` 判断是否到间隔，到则 `Save(CurrentSlotId)` |
| **退出回主菜单前** | 暂停菜单点「退出游戏」确认后 | `PausePanel` 退出逻辑里先 `SaveManager.Save(CurrentSlotId)` 再 `LoadAsync("MainMenuScene")` |
| **新建游戏后** | `GameBootstrap.InitializeScene` NewGame 分支末尾 | `SaveManager.Save(config.selectedSlotId)` 存初始档 |

### `SaveManager` 需要加的字段/方法

```csharp
public class SaveManager : Singleton<SaveManager>
{
    // 当前活跃槽位（NewGame 时设为 selectedSlotId，ContinueGame 时设为 LoadSlotId）
    public string CurrentSlotId { get; private set; }

    // 自动存档配置（从 settings.json 读取，全局）
    public bool AutoSaveEnabled { get; set; } = true;
    public int AutoSaveIntervalDays { get; set; } = 3;

    // 内部计数：距上次自动存档过了几天
    private int _daysSinceLastAutoSave;

    // 订阅 TimeDayChangedEvent
    private void OnDayChanged(TimeDayChangedEvent evt)
    {
        if (!AutoSaveEnabled || string.IsNullOrEmpty(CurrentSlotId)) return;

        _daysSinceLastAutoSave++;
        if (_daysSinceLastAutoSave >= AutoSaveIntervalDays)
        {
            Save(CurrentSlotId);
            _daysSinceLastAutoSave = 0;
        }
    }

    // Save 和 Load 里更新 CurrentSlotId
    public bool Save(string slotId)
    {
        CurrentSlotId = slotId;  // ← 新增
        // ... 原有逻辑 ...
    }

    public bool Load(string slotId)
    {
        CurrentSlotId = slotId;  // ← 新增
        _daysSinceLastAutoSave = 0;  // 读档后重置计数
        // ... 原有逻辑 ...
    }
}
```

### 全局设置存储（settings.json）

自动存档设置是全局的（不分槽位），独立存一个文件：

```csharp
// 路径：Application.persistentDataPath/settings.json
[Serializable]
public class GameSettings
{
    public bool autoSaveEnabled = true;
    public int autoSaveIntervalDays = 3;
}
```

`SaveManager` 启动时读 `settings.json`，设置面板修改时写回。不随游戏存档走。
